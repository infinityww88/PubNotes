官方文档

```
从我们的测试结果上看，极限情况下，即xLua(Lua53)使用wrapper，PuerTS(v8_9.4)使用xIl2cpp模式同时生成wrapper：

在安卓上，普洱的跨语言性能是xlua的2倍左右
```

## PuerTS 跨语言调用效率

**PuerTS 的 C# ↔ JS 跨语言调用本身很快（尤其开启静态绑定后），但"每帧、每个对象、反复穿越"仍然不推荐。**  
正确做法是：**C# 每帧只调一次 JS 的 Update 调度器，对象逻辑全在 JS 内部跑；避免在 Update 里逐对象逐属性跨语言交互。**

### ✅ 单次调用开销很低

- PuerTS 通过 V8/QuickJS 绑定 C# 反射或**静态胶水代码（Wrapper）**，C# ↔ JS 函数调用本身开销在微秒级
- 开启 **Generate Wrapper（静态绑定）** 后，复杂对象传递比反射快约 2 倍，接近 Lua

### ❌ 真正贵的地方
- **V8 ↔ Mono/IL2CPP 上下文切换**（每次调用都有）
- **值类型序列化**：`Vector3` / `Quaternion` 传参需拆箱/装箱或 Marshal
- **高频次数放大**：N 个对象 × 60fps = 巨量跨语言调用 → 明显掉帧

## 二、在 Update 里直接互调 — 为什么不推荐？

❌ **不推荐写法（反例）**

```csharp
// C# MonoBehaviour.Update
void Update() {
    jsUpdate?.Invoke();  // 每个 GameObject 都在自己的 Update 中调用 js
    // 或者更糟糕的，每个对象每帧在 js 中读取 CS.UnityEngine.Transform.get_position()
}
```

问题：
- 若你有 200 个实体 = 每帧 200 次 C#→JS 穿越
- JS 每帧读 `transform.position` = 再叠加 N 次 JS→C#
- 移动端/小游戏会明显掉帧

## 三、推荐的 Update 交互模式 ✅

### ✅ 模式 1：JS 侧自己维护 Update 调度器（官方推荐）

**C# 侧 — 只调一次**

JS 中设置一个全局的 Tick（Update）回调，所有 js 对象都在这其中注册自己的 Update 回调，然后每次帧到来时，由 C# 全局调用一次 JS 的 Tick，剩下的都在 js 中执行，尽量减少 C#-JS 的穿越。

```csharp
public class JsBridge : MonoBehaviour {
    void Update() {
        jsEnv.Global.GetAction("Tick")?.Invoke(Time.deltaTime);
    }
}
```

**TS 侧 — 统一管理所有逻辑对象**
```ts
const tickList: Array<(dt: number) => void> = [];

export function Tick(dt: number) {
    for (let i = 0; i < tickList.length; i++) {
        tickList[i](dt);
    }
}

export function registerTick(fn: (dt: number) => void) {
    tickList.push(fn);
}
```

✅ 优点：
- C#→JS 每帧只 **1 次** 跨语言调用
- 所有实体 Update 在 JS 内循环，无额外开销

### ✅ 模式 2：逻辑全放 JS，C# 只负责渲染 / 输入

- JS 算好结果 → 一次性写回 C#（`transform.position = v3`）
- 或 JS 自己持有位置，C# 只读（避免每帧 getter 跨语言）

### ✅ 模式 3：生成静态 Wrapper（必做，性能提升显著）

在 Unity Editor：
```
Tools → PuerTS → Generate For Il2Cpp Mode (All in One)
```
可减少类型转换开销 40%~60%

## 四、什么时候可以放心跨语言每帧？

| 场景 | 是否 OK |
|----|----|
| C# 每帧调 **1 次** JS 调度器 | ✅ 推荐 |
| JS 内部每帧算 Vector / 状态 | ✅ 推荐 |
| 少量（1~几个）特殊对象每帧调 JS | ⚠️ 可接受 |
| N 个对象每帧各调一次 JS | ❌ 不推荐 |
| JS 每帧读 `transform.position` / `rotation` | ❌ 尽量避免 |

## 五、一句话总结

> **PuerTS 跨语言调用本身不慢，慢的是"高频穿越"。**  
> ✅ Update 里只做一次 C#→JS 入口调用，实体逻辑、Tick 分发全放 TS 里跑。  
> ✅ 开启静态 Wrapper，避免每帧传值类型（Vector3/Quaternion）来回穿越。

# 每次函数穿越调用都涉及参数的打包解包

是的，**在绝大多数情况下，JS 调用 C# 函数时，每一次调用都会涉及“参数打包 + 解包（marshal）”**。  
但**开销大小取决于参数类型和是否开启静态绑定（Wrapper）**。

## 一、结论先行 ✅

| 调用场景 | 是否打包 / 解包 | 成本 |
|----|----|----|
| 基本类型（number / boolean / string） | ✅ 有 | 极低 |
| struct（Vector3 / Quaternion） | ✅ 有 | 中等 |
| class / object | ✅ 有 | 低（传引用） |
| delegate / Action | ✅ 有 | 中等 |
| 数组 / List | ✅ 有 | 较高 |
| 开启 **Generate Wrapper** | ✅ 仍有，但更快 | ✅ 推荐 |
| 反射模式（未生成 Wrapper） | ✅ 有 + 反射 | ❌ 慢 |

**不存在“零成本”的跨语言调用**

## 二、JS → C# 调用的真实过程（简化版）

```text
JS 调用
 ↓
V8 / QuickJS
 ↓ 参数打包（Marshal）
 ↓
C# 接收参数（Unmarshal）
 ↓
执行 C# 方法
 ↓
返回值再次打包 → JS
```

### 示例
```ts
CS.UnityEngine.Debug.Log("hello");
```

实际发生：
1. `"hello"` 从 JS string → C# string（分配内存）
2. 调用 `Debug.Log(string)`
3. 无返回值（否则还要反向 Marshal）

## 三、不同类型参数发生了什么？

### ✅ 基本类型（快）
| JS | C# | 行为 |
|----|----|----|
| number | int / float | 值拷贝 |
| boolean | bool | 值拷贝 |
| string | string | 拷贝（UTF16 → UTF8） |

✅ 开销极小，可放心高频使用

### ⚠️ struct（Vector3 / Color / Quaternion）
```ts
CS.UnityEngine.Transform.position = new CS.UnityEngine.Vector3(1,2,3);
```

会发生：
- JS 创建临时 struct
- 字段逐个拷贝
- C# 侧构造新 struct

❗ **每帧大量 Vector3 跨语言调用 = 性能杀手**

✅ 优化方式：
- 批量计算
- 一次性传数组
- 逻辑留在 JS

### ⚠️ class / object（相对便宜）
```ts
CS.GameLogic.Player.SetTarget(obj);
```

- 只传 **C# 对象指针**
- 不复制数据
- 后续访问仍可能产生跨语言调用

✅ 成本较低，但仍不是零

### ⚠️ delegate / Action
```ts
CS.EventManager.OnTick = (dt: number) => {};
```

- JS 函数 → C# delegate
- 需要包装 + 生命周期管理
- 调用时双向 Marshal

✅ 适合“低频事件”
❌ 不适合每帧 N 次调用

## 四、静态绑定（Wrapper）能消除打包吗？

❌ **不能消除**
✅ **大幅降低开销**

| 模式 | 是否打包 | 额外成本 |
|----|----|----|
| 反射 | ✅ | 反射 + Marshal |
| 静态绑定 | ✅ | 仅 Marshal |
| 原生 C# 调用 | ❌ | 无 |

**静态绑定 ≈ 减少 30%–60% 跨语言成本**

> **是的，JS 调用 C# 函数，每次都会进行参数打包 / 解包（Marshal）。**  
> - 基本类型：几乎可以忽略  
> - struct / 数组：有明显成本

## 每一帧，应用程序整体调用 JS 一次或固定少量的几次

> **“C# 每帧调 1 次 JS 调度器”**  
> 👉 **指整个程序（或整个系统）只在 `MonoBehaviour.Update` 里调用一次 JS**  
> ❌ **不是每个 GameObject / 每个对象各自调用一次 JS**

## 用一句话再强调一遍

> **C# 侧：每帧只做一件事 —— 通知 JS“这一帧到了”**  
> **JS 侧：自己决定有多少个对象、谁需要 Tick、怎么 Tick**

## 正确示例 ✅（推荐）

### C#（只有一个 Update）

```csharp
public class JsMain : MonoBehaviour
{
    void Update()
    {
        // ✅ 整个游戏每帧只调用一次 JS
        JsEnv.Eval("CS.Tick && CS.Tick(UnityEngine.Time.deltaTime)");
    }
}
```

### JS / TS（JS 内部调度）

```ts
const systems: ((dt: number) => void)[] = [];

export function Tick(dt: number) {
    for (const system of systems) {
        system(dt);
    }
}

export function registerSystem(fn: (dt: number) => void) {
    systems.push(fn);
}
```

```ts
registerSystem((dt) => {
    // 角色系统
});

registerSystem((dt) => {
    // AI 系统
});
```

✅ **无论多少对象，C# → JS 永远是 1 次 / 帧**

## JS 中的向量矩阵运算

uerTS + TS 做游戏逻辑时**最核心的性能痛点** 👍  

> ✅ **向量 / 矩阵 / 四元数运算，坚决不要在 JS 里反复调 `CS.UnityEngine.Vector3 / Mathf / Matrix4x4`**  
> ✅ **正确做法：JS 层用纯 TS 向量类或 gl-matrix 做全部运算，只在"最终结果"写回 C# Transform/Rigidbody 时跨一次语言边界**

## 一、为什么在 JS 调 C# 数学库是性能雷区

```ts
// ❌ 每帧几百次 Vector3.Dot / Distance / Normalize → 数百次 C#↔JS Marshal
const dir = CS.UnityEngine.Vector3.Normalize(target.sub(pos));
```

问题：
- **每个操作符 = 1 次跨语言调用 + struct 打包/拆包**
- `Vector3` 是 struct，JS 每拿到/传出都要新建 C# struct
- 向量运算密集时 → 跨语言调用次数爆炸 💥

实测量级（参考）：
| 操作 | 单次耗时 | GC |
|----|----|----|
| JS 内部 vec3.dot | ~2μs | 0 |
| `CS.Vector3.Dot()` | ~0.1~0.15ms | 有 |
→ 差 **50~100 倍**

## 二、✅ 正确方案：JS 层做数学，只同步结果给 C#

### 架构原则

```
每帧：
JS Tick
 ├─ JS 向量/矩阵运算（纯 TS / gl-matrix）← 零跨语言
 └─ 最终结果 → C# transform.position / rotation（1 次写回）
```

❌ 不要在 JS 里：
- `CS.Vector3.Add / Subtract / Dot / Cross / Normalize`
- `CS.Mathf.Sin / Clamp / Lerp`（简单 Math 直接用 JS Math）

## 三、落地方案 1：手写轻量 Vec3 / Mat4（最常见）

```ts
// vec3.ts
export class Vec3 {
    constructor(public x = 0, public y = 0, public z = 0) {}

    add(b: Vec3)      { return new Vec3(this.x+b.x, this.y+b.y, this.z+b.z); }
    sub(b: Vec3)      { return new Vec3(this.x-b.x, this.y-b.y, this.z-b.z); }
    dot(b: Vec3)      { return this.x*b.x + this.y*b.y + this.z*b.z; }
    scale(s: number)  { return new Vec3(this.x*s, this.y*s, this.z*s); }
    length()          { return Math.hypot(this.x, this.y, this.z); }
    normalize()       { const l = this.length(); return l < 1e-6 ? new Vec3() : this.scale(1/l); }
}
```

游戏逻辑全用这个类，**完全不碰 C# Vector3**。

## 生成静态 Wrapper（静态绑定）

**“生成静态 Wrapper（静态绑定）”是 PuerTS 性能优化的核心机制之一**。  

**生成静态 Wrapper = 为 C# 类 / 方法 / 属性提前生成“C++/C# 胶水代码”，让 JS 调用 C# 时不再走反射，而是直接调用。**

## 二、没有 Wrapper 时发生了什么？（慢）

### ❌ 反射模式（默认）

```ts
CS.UnityEngine.Debug.Log("hello");
```

内部流程：

```
JS
 ↓
PuerTS
 ↓
C# Reflection
   ├─ 查 MethodInfo
   ├─ 参数装箱
   ├─ MethodBase.Invoke
 ↓
执行 Debug.Log
```

问题：
- **反射很慢**
- 每次调用都要查元数据
- IL2CPP 下反射更慢
- 参数 struct 还会反复 Marshal

## 三、生成静态 Wrapper 后发生了什么？（快）

### ✅ 生成后的调用链

```
JS
 ↓
PuerTS
 ↓
生成的静态胶水函数（Wrapper）
 ↓
直接调用 C# 方法（无反射）
```

✅ 类似你自己手写：

```csharp
// 自动生成的代码（示意）
static void Debug_Log(JSValue arg0) {
    string s = JSMarshal.ToCSharp<string>(arg0);
    UnityEngine.Debug.Log(s);
}
```

➡ **零反射、路径最短**


## 四、Wrapper 的本质是什么？

### ✅ 本质 = **强类型的桥接函数**

| 层 | 内容 |
|----|----|
| JS | 调用 `CS.UnityEngine.Debug.Log` |
| Wrapper | 把 JS 值 → C# 值 |
| C# | 直接调用目标方法 |

Wrapper 做了三件事：
1. **参数 Marshal（JS → C#）**
2. **直接调用 C# 方法**
3. **返回值 Marshal（C# → JS）**

## 五、Wrapper 长什么样（简化示例）

### C# 类

```csharp
namespace GameLogic {
    public class Player {
        public int Hp;
        public void Damage(int dmg) {
            Hp -= dmg;
        }
    }
}
```

### 生成的 Wrapper（示意）

```csharp
// 自动生成
public static class Player_Wrapper {
    public static void Damage(IntPtr ctx) {
        var self = JSHelper.GetSelf<Player>(ctx);
        int dmg = JSHelper.GetArg<int>(ctx, 0);
        self.Damage(dmg);
    }
}
```

✅ JS 调用时：
```ts
player.Damage(10);
```

✅ 实际执行的是：
```csharp
Player_Wrapper.Damage(ctx);
```

## 六、为什么叫「静态」Wrapper？

因为它是：
- ✅ **编译期生成**
- ✅ **方法地址固定**
- ✅ **不需要运行时反射查找**
- ✅ **IL2CPP 友好**

对比：

| 模式 | 调用方式 |
|----|----|
| 反射 | 运行时查表 |
| 静态 Wrapper | 编译期硬编码调用 |

## 七、性能差距有多大？

| 场景 | 反射 | 静态 Wrapper |
|----|----|----|

| struct 参数（Vector3） | 慢 | ✅ 明显快 |
| IL2CPP | 很慢 | ✅ 可接受 |
| 高频 Update | ❌ 不行 | ✅ 可行 |

📌 **不是零成本，但接近原生 C# 调用的一半以上性能**

## 十一、一句话总结 ✅

**生成静态 Wrapper = 把“反射调用”变成“提前写好的强类型桥接函数”，让 JS 调用 C# 从“查字典”变成“直接打电话”。**

总之尽可能少的进行 C#-JS 穿越调用。能在 C# 中执行 Javascript，不意味着你应该大量频繁的进行交互，相反应该做的是，尽可能让两边隔离、少交互，表现地就像两个独立的环境一样，只在必要的时刻才进行交互，只要遵循这个原则，就可以最少的遇到性能瓶颈。每帧少量固定次数的调用可以。尽可能将固定的逻辑写在 C# 一侧，JS 只作为逻辑胶水层。
